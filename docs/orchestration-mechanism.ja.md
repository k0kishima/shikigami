# オーケストレーションの実現方法

<p align='center'>
  <a href='./orchestration-mechanism.md'>English</a> | 日本語
</p>

このドキュメントでは、[オーケストレーションの目的](./purpose-of-orchestration.ja.md)で述べた概念を Shikigami がどのように実現しているかを説明します。

## 技術基盤

Shikigami のオーケストレーションは [Claude Code](https://claude.ai/code) の **Agent Teams** 機能を基盤としています。Agent Teams は Claude Code のセッションから複数のサブエージェント（「チームメイト」）を生成し、それぞれに独自のシステムプロンプトとコンテキストを持たせる機能です。Shikigami はこの機能を利用して、ユーザーの要件に応じたタスクフォースを動的に生成します。

## システム構成要素

オーケストレーションの仕組みは4つの主要コンポーネントで構成されています。

| コンポーネント | ファイル | 役割 |
|---|---|---|
| CLI | `bin/shikigami` | オーケストレーターを起動するエントリーポイント |
| オーケストレーター指示書 | `CLAUDE.md` | 必須ワークフローを定義 |
| オーケストレーターロールテンプレート | `roles/orchestrator.md` | オーケストレーターのペルソナと設計原則を定義 |
| スペシャリストロールテンプレート | `roles/*.md` | 生成されるエージェントのペルソナを定義 |

## 起動フロー

ユーザーが `shikigami` コマンドを実行すると、CLI スクリプトは以下の手順を実行します。

1. **前提条件の確認** — Claude Code がインストールされているか検証
2. **機能の有効化** — `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` を設定して Agent Teams を有効化
3. **システムプロンプトの構築** — オーケストレーターのロールテンプレート（`roles/orchestrator.md`）を読み込み、`roles/` 内の利用可能なロールテンプレートを列挙し、以下を含むシステムプロンプトを組み立てる:
   - オーケストレーターのロール定義（ペルソナ、temperature、設計原則）
   - 利用可能なロール一覧とファイルパス
   - ロールテンプレートファイルを読み込んでチームメイトを生成する方法の指示
4. **起動** — `claude --system-prompt "$SYSTEM_PROMPT"` を実行し、Claude Code をオーケストレーターとして起動

## ワークフローの強制

オーケストレーターの振る舞いは `CLAUDE.md` によって制御されており、厳格な5ステップのワークフローが強制されます。

1. **要件分析** — オーケストレーターがユーザーに質問し、スコープを確認する
2. **タスクフォース提案** — 定義済みのロールテンプレートのみを使用してチーム構成を提案し、ユーザーの承認を得る
3. **タスクフォース生成** — ロールテンプレートファイルを読み込み、その内容を指示としてエージェントを生成する
4. **編成報告** — 生成されたタスクフォースの構成をユーザーに表示する
5. **統制** — タスクの割り当て、フィードバックループの管理、結果の報告を行う

`CLAUDE.md` は禁止行動（例：要件分析なしに作業を開始する、実装作業を直接行うなど）も定義しており、オーケストレーターがコーディネーターとしての役割に留まることを保証します。

## ロールテンプレートとエージェント生成

`roles/` 内の各ロールテンプレートは、以下の要素で専門エージェントのペルソナを定義します。

- **バックストーリー** — エージェントの振る舞いを形づける架空の経歴
- **Temperature** — そのロールにおける創造性と確定性のトレードオフを制御
- **責務と制約** — エージェントがすべきこと、してはならないこと

オーケストレーターがタスクフォースを生成する際、必要な各ロールのテンプレートファイルを読み込み、その内容を Agent Teams 経由でサブエージェントのシステムプロンプトとして渡します。これにより、各エージェントはオーケストレーターのコンテキストから独立した専門指示のもとで動作します。

### 利用可能なロール

| ロール | Temperature | 目的 |
|---|---|---|
| Coder | 0.3 | 要件・設計仕様に基づく実装 |
| Reviewer | 0.2 | コード品質と設計の評価 |
| Tester | 0.3 | テスト実行と追加テストケースの作成 |
| Security Engineer | 0.2 | セキュリティ脆弱性の検出 |
| Performance Engineer | 0.2 | パフォーマンス分析とボトルネックの特定 |
| System Architect | 0.4 | 技術選定とシステム設計 |

## 並列化とレイヤー化の実現

[オーケストレーションの目的](./purpose-of-orchestration.ja.md)で述べた概念は、以下のように実現されています。

### 並列化

同一ロールの複数エージェント（例：Coder 2名）を生成し、同じ問題に独立して取り組ませることが可能です。それぞれの出力が、目的ドキュメントで述べた解の多様性を提供します。

### レイヤー化

異なるロールが自然にレイヤーを形成します。例えば：
- **生成レイヤー**: Coder エージェントが並列で実装を生成
- **評価レイヤー**: Reviewer エージェントが実装を比較・評価

オーケストレーターは要件に基づいてこれらのレイヤーを動的に設計します。固定されたレイヤー構造は存在しません。

### レイヤーごとの Temperature

ロールテンプレートの temperature 設定はレイヤー設計の原則を反映しています。
- 探索ロール（Coder, System Architect）は多様性のため高い temperature（0.3〜0.4）
- 収束ロール（Reviewer, Security Engineer, Performance Engineer）は安定した評価のため低い temperature（0.2）

## アーキテクチャ概観

```
ユーザー
 ↓
bin/shikigami (CLI)
 │  Agent Teams を有効化
 │  ロールテンプレートからシステムプロンプトを構築
 ↓
Claude Code (オーケストレーター)
 │  システムプロンプト: orchestrator.md + ロール一覧
 │  ワークフロー強制: CLAUDE.md
 │
 │  Step 1-2: 要件分析 → タスクフォース提案
 │  Step 3:   Agent Teams でエージェント生成
 ↓
┌─────────┬─────────┬─────────┐
│ Coder   │ Coder   │ Tester  │  ← 生成・実行レイヤー
└────┬────┴────┬────┴────┬────┘
     │         │         │
     ↓         ↓         ↓
┌─────────────────────────────┐
│ Reviewer                    │  ← 評価・収束レイヤー
└─────────────────────────────┘
```

具体的な構成はタスクごとに異なります。オーケストレーターが要件分析後に最適な構造を決定します。
