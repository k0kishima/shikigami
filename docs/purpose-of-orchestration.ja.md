# オーケストレーションの目的

<p align='center'>
  <a href='./purpose-of-orchestration.md'>English</a> | 日本語
</p>

AIエージェントをオーケストレーションする目的は、LLMの非決定性を克服し、人間の介入を最小化しながら高品質な出力を得ることです。  
そのために並列化と階層化を採用しています。

## 現行の LLM がガチャであることへは並列化で単純に対応

### AIの非決定的な性質

- 現行の LLM は、関数のように入力に対して単一の決定値を返すものではない
- 具体的には、同じモデルに同じプロンプトを(さらには同一 temperature を)与えても、内部のサンプリングやトークン選択の揺らぎにより、生成結果は毎回異なり得る
  - つまり、 LLM の応答は 「入力 → 出力」ではなく「入力 → 出力の確率分布」である
- このような性質から「AIはガチャ」と比喩されたりもする

### 並列実行というソリューション

- 非決定的な生成モデルにおいては、局所的に存在する「当たり解」を単発試行で得られないことは珍しくない
- この問題に対しては、サンプリング数（試行回数）を増やすという単純かつ有効な戦略がある
- 並列実行はウォール・タイム（人間が実際に待つ時間）を増やさずに出力の多様性を確保する実装上の解である

### 結論

- 現行の LLM は、本質的に非決定的な応答生成を行う
- そのため、逐次的に1回ずつ試行するよりも、同時に複数のサンプルを生成した方が期待値的に有利である
- この「同時に複数サンプルを取得する」ための実装手段として並列実行を採用する

## 並列化の限界を補うために上位レイヤーを設置する

### 並列化だけでは不十分

- 並列化は単に同じ問題を独立に n 回サンプリングするということをしているだけ
- 「量」は増えるが「質」は自動では上がらない
- 並列化するだけだと、結局人間が増えた応答を全部読んで、その中から最適解を探すという手間がかかる

### 並列化を収束させる上位レイヤーの設置

- 並列化それ自体は、生成結果に対する「相対的な評価」や「選別」を一切しない
- 並列実行のレイヤーの1つ上にメタ評価をするレイヤーを配置することでこの問題に対応
- 複数案を俯瞰し、比較・統合・棄却を行うことで、並列レイヤーのアウトプットを「使える解」に収束させる

### 結論

- 並列化で解の多様性を確保し、その上のレイヤーで効率的な収束を行うが、これは評価レイヤーである
- 評価と意思決定は別タスクである
  - 評価は「各案の良し悪しを判定する」ことであり、意思決定は「どの案を採用するか決める」ことである
  - これらを分離することで、評価基準の一貫性を保ちつつ、文脈に応じた柔軟な意思決定が可能になる
- このような考え方に基づき、例えば評価レイヤーの上に意思決定レイヤーを配置するという構成が考えられる
- ただし、これはレイヤー分けの原則であり、実際の階層構造は要件に応じて動的に設計される

## 階層化によって意思決定までを自動化する

### 要件に応じた動的な階層設計

- 階層化における層の数や各層の役割は、タスクの性質によって異なるべきである
- 事前に固定された階層構造を適用するのではなく、要件を分析した上で最適な階層構造を設計する必要がある
- Shikigami では、オーケストレーターが要件分析を行い、そのタスクに最適な構成を持つタスクフォースを動的に生成する
  - 単純なタスク: 浅い階層（例: 生成 → 意思決定の2層）
  - 標準的なタスク: 中程度の階層（例: 生成 → 評価 → 意思決定の3層）
  - 複雑なタスク: 深い階層（例: 調査 → 生成 → 評価 → 統合 → 意思決定の5層）
- このように階層構造そのものを要件に応じて最適化することで、過剰な複雑さを避けつつ必要な品質を確保する

### 階層設計における temperature の原則

- オーケストレーターが設計した階層において、各レイヤーの temperature は役割に応じて設定される
- temperature は創造性（多様性）と確定性のトレードオフを制御するパラメータである
- 探索を担うレイヤーほど temperature が高く、確率分布は平坦になる
  - 多様性や創造性を重視するため
- 収束を担うレイヤーほど temperature が低く、確率分布は鋭くなる
  - 安定した意思決定を行うため

### 結論

- 階層化は役割分離と温度勾配によって効率と精度を向上させる
- ただし、最適な階層構造はタスクごとに異なる
- Shikigami のオーケストレーターは要件分析に基づいて最適な階層構造を動的に設計し、タスクフォースとして実現する
