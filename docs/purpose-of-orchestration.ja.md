# オーケストレーションの目的

<p align='center'>
  <a href='./purpose-of-orchestration.md'>English</a> | 日本語
</p>

AIエージェントをオーケストレーションする目的は、LLMの非決定性を克服し、人間の介入を最小化しながら高品質な出力を得ることです。  
そのために並列化とレイヤー化を採用しています。

## 並列化による解の多様性の確保

### AIの非決定的な性質

- 現行の LLM は、関数のように入力に対して単一の決定値を返すものではない
- 具体的には、同じモデルに同じプロンプトを(さらには同一 temperature を)与えても、内部のサンプリングやトークン選択の揺らぎにより、生成結果は毎回異なり得る
  - つまり、 LLM の応答は 「入力 → 出力」ではなく「入力 → 出力の確率分布」である
- このような性質から「AIはガチャ」と比喩されたりもする

### 並列実行というソリューション

- 非決定的な生成モデルにおいては、局所的に存在する「当たり解」を単発試行で得られないことは珍しくない
- この問題に対しては、サンプリング数（試行回数）を増やすという単純かつ有効な戦略がある
- 並列実行はウォール・タイム（人間が実際に待つ時間）を増やさずに出力の多様性を確保する実装上の解である

### まとめ

- 現行の LLM は、本質的に非決定的な応答生成を行う
- そのため、逐次的に1回ずつ試行するよりも、同時に複数のサンプルを生成した方が期待値的に有利である
- この「同時に複数サンプルを取得する」ための実装手段として並列実行を採用する

## レイヤー化による収束とアウトプットの最適化

### 並列化の限界

- 並列化は単に同じ問題を独立に n 回サンプリングするということをしているだけ
- 並列化それ自体は、生成結果に対する「相対的な評価」や「選別」を一切しない
  - これ単体で運用しても、結局ユーザーが増えた応答を全部読んで、その中から最適解を探すという手間が必要である
- つまり、「量」は増えるが「質」は自動では上がらない

### 並列化を収束させる上位レイヤーの設置

- 並列実行のレイヤーの1つ上にメタ評価をするレイヤーを配置することで前節の問題に対応
- 上位レイヤーは下位レイヤーが提出した複数案を俯瞰し、比較・統合・棄却を行うことで、アウトプットを「使える解」に収束させる

### 要件に応じた動的なレイヤー設計

- レイヤーは単に並列化を収束させる目的でのみ切り分けるものではない
- 評価や意思決定などの役割でも分離しうる
- 事前に固定されたレイヤー構造を適用するのではなく、要件を分析した上で最適なレイヤー構造を設計する必要がある
- Shikigami では、オーケストレーターと呼ばれるロールの AI エージェントをユーザーとのインターフェースとして設置する
  - オーケストレーターはユーザーのプロンプトをベースに要件分析を行い、要件定義結果に対して最適な構成を持つタスクフォースを動的に生成する

### レイヤー設計における temperature の原則

- オーケストレーターが設計した各レイヤーの temperature[^1] は役割に応じて設定される
- 探索を担うレイヤーほど temperature が高く、確率分布は平坦になる
  - 多様性や創造性を重視するため
- 収束を担うレイヤーほど temperature が低く、確率分布は鋭くなる
  - 安定した意思決定を行うため

### まとめ

- 事前に固定されたレイヤー構造を適用するのではなく、要件を分析した上で最適なレイヤー構造を設計して適用する
  - このことがリソース削減や、アウトプットの質を担保することにつながる
- ユーザーは最終的なアウトプットのみ確認し、各レイヤーでの途中経過に介在する必要はない

[^1]: temperature は創造性（多様性）と確定性のトレードオフを制御するパラメータである。
