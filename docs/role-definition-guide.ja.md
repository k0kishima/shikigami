# ロール定義ガイド

<p align='center'>
  <a href='./role-definition-guide.md'>English</a> | 日本語
</p>

このドキュメントでは、Shikigamiで使用するロールテンプレートの定義方法を説明する。

## 概要

ロールテンプレートは `roles/` ディレクトリに配置する。ファイル名は `{role_name}.md` の形式とする。

## 記載言語

ロールテンプレートは英語で記載すること。

ロールテンプレートは人間ではなくAIエージェントが読み込み解釈する。現在のLLMは一般的に英語での理解・推論精度が高いため、英語で記載することでAIが指示を理解し従う能力を最大化できる。

人間向けのドキュメント（`docs/` 内のファイルなど）は複数言語で提供しても良いが、ロールテンプレートは常に英語とする。

## 書式の由来

本書式は、マルチエージェントフレームワークである CrewAI のエージェント定義をベースに設計した。

| 項目 | 由来 |
|------|------|
| Description | CrewAI の `role` に相当 |
| Goal | CrewAI |
| Backstory | CrewAI |
| Constraints | MetaGPT の `constraints` |
| Tools | CrewAI |
| Temperature | LLM設定として独自追加 |

CrewAI を選定した理由は、シンプルで可読性が高く、Markdown形式との親和性が高いため。

## 基本構造

すべてのロールテンプレートは以下の項目を含む。

| 項目 | 必須 | 説明 |
|------|------|------|
| Description | ○ | ロールの概要 |
| Goal | ○ | 達成すべき目標 |
| Backstory | ○ | ペルソナ設定 |
| Constraints | ○ | 制約・禁止事項 |
| Tools | ○ | 使用可能なツール |
| Temperature | ○ | 推奨temperatureと理由 |

## 各項目の記載ガイドライン

### Description

ロールの概要を1〜2文で記載する。タスクフォース内での位置づけ（どのレイヤーに配置されるかなど）があれば含める。

```markdown
## Description

ソフトウェア開発においてコードの実装を担当するロール。要件や設計に基づいて、高品質で保守性の高いコードを生成する。
```

### Goal

このロールが達成すべき目標を箇条書きで記載する。具体的かつ検証可能な目標が望ましい。

```markdown
## Goal

- 与えられた要件を満たすコードを正確に実装する
- 可読性が高く、保守しやすいコードを書く
- 既存のコードベースのスタイルや規約に従う
```

### Backstory

ロールのペルソナ設定を記載する。専門性の背景や経験を含めることで、ロールの振る舞いに一貫性を持たせる。

```markdown
## Backstory

10年以上の実務経験を持つシニアソフトウェアエンジニア。複数の言語とフレームワークに精通し、クリーンコードの原則とベストプラクティスを重視する。
```

### Constraints

ロールが守るべき制約や禁止事項を箇条書きで記載する。

```markdown
## Constraints

- テスト可能な設計を心がける
- 過度な抽象化や早すぎる最適化を避ける
- 要件にない機能を勝手に追加しない
```

### Tools

ロールが使用可能なツールを箇条書きで記載する。

```markdown
## Tools

- ファイルの読み書き
- コマンド実行（ビルド、テスト、リンター等）
- Web検索（ドキュメント参照）
```

### Temperature

推奨するtemperature値と、その理由を記載する。

```markdown
## Temperature

0.3

安定性と一貫性を重視するため低めに設定。コード生成では創造性よりも正確性が求められる。
```

temperatureの目安：

| 値 | 用途 |
|----|------|
| 0.1〜0.3 | 正確性・安定性重視（評価、検証、コード生成） |
| 0.4〜0.6 | バランス型（要件分析、設計） |
| 0.7〜1.0 | 創造性・多様性重視（アイデア出し、探索） |

## 特別なロールの追加項目

### オーケストレーター

オーケストレーターはタスクフォースを設計・生成する責務を持つため、基本項目に加えて「タスクフォース設計の原則」セクションを含める。

```markdown
## タスクフォース設計の原則

オーケストレーターはタスクフォースを設計する際、以下の原則に従う。

### 並列化の判断基準
...

### レイヤー構成の判断基準
...
```
